1. System Architecture Diagram and Design Rationale

1.1 Architecture Diagram

Figure 1 — Conversion Engine architecture. Arrows show directional data flow.

1.2 Design Rationale

Research-First, Not Spray-and-Pray. The pipeline begins with five enrichment steps before the agent writes a single word of outreach. The design decision: an LLM composing outreach without a grounded brief produces generic emails that get ignored. Every claim in the sent email must trace back to a field in hiring_signal_brief.json.

Segment-Driven Routing. Prospects are classified into four ICP segments (Series A/B funding, mid-market restructure, leadership transition, specialized capability gap) plus an abstain state. Each segment maps to a dedicated email angle. If segment confidence falls below 0.6, the agent is forced to send a generic exploratory email rather than assert a pitch it cannot support. This is enforced in Python post-LLM, not delegated to the model.

Honesty Constraints Enforced in Code. Three conditions trigger protective overrides: (1) velocity_label == "insufficient_signal" forces "ask" language, not assertions; (2) segment_confidence < 0.6 routes to abstain; (3) bench_available == False flags a capacity gap and routes to a human, preventing false capacity promises.

Kill Switch Before Any Outbound. KILL_SWITCH_LIVE_OUTBOUND=false (the safe default) redirects all email and SMS to configurable sink addresses. No prospect is ever contacted without an explicit environment change and program-staff approval.

3. Enrichment Pipeline Documentation

The pipeline runs five signals in sequence, assembles a hiring_signal_brief.json, and validates it against the JSON Schema before the agent sees any data. The sample values below are from the Stripe test run (generated_at: 2026-04-23T16:30:22Z).

Signal 1 — Crunchbase ODM Firmographics

Source: agent/enrichment/crunchbase_enricher.py — fuzzy-match against local Crunchbase CSV using thefuzz (token sort ratio ≥ 75).

Output fields: name, domain, industry, employee_count, funding_total, last_funding_stage, last_funding_date, location, description, crunchbase_id

funding_total: "$6.5B" · last_funding_stage: "debt" · employee_count: "8000+" · industry: "Financial Services"

Classification link: last_funding_stage determines Segment 1 (Series A/B) or Segment 2 (mid-market). employee_count is parsed to an integer and combined with funding stage to score segment confidence. funding_event.confidence is 0.80 when both stage and funding total are present, 0.50 with stage only.

Signal 2 — Job-Post Velocity Scraping

Source: agent/enrichment/job_post_scraper.py — Playwright headless browser scrapes company careers pages and Wellfound. Graceful fallback: returns {"status": "no_data"} on timeout or block.

Output fields: open_roles_today (int), role_titles (list[str]), sources (list), status

open_roles_today: 0 · role_titles: [] · status: "no_data" (stripe.com blocks scraper)

Classification link: open_roles_today feeds hiring_velocity.velocity_label (insufficient_signal / tripled_or_more / …). AI-adjacent role count (regex on titles) feeds Segment 4 scoring. Low role count triggers weak_hiring_velocity_signal honesty flag, forcing ask-not-assert email language.

Signal 3 — Layoffs.fyi Integration

Source: agent/enrichment/layoffs_enricher.py — fuzzy-match against data/layoffs_fyi.csv (same token sort ratio threshold).

Output fields: detected (bool), date, headcount_reduction (int), percentage_cut (float), source_url

detected: true · date: "2025-01-21" · headcount_reduction: 300 · percentage_cut: 0.03 · source_url: "https://www.businessinsider.com/stripe-layoffs-staff-payments-memo-2025-1"

Classification link: A detected layoff within 120 days + Series A/B or mid-market profile → Segment 2 (restructure). layoff_event.confidence is 0.90 when percentage_cut or headcount_reduction is present (quantified), 0.70 for date-only. Triggers layoff_overrides_funding honesty flag when layoff coincides with Series A/B funding.

Signal 4 — Leadership Change Detection (LLM)

Source: agent/enrichment/pipeline.py :: _detect_leadership_change_llm() — Qwen3 via OpenRouter. Runs after job scraper; uses company name, industry, and observed role titles as context. Instructed to infer rather than fabricate.

Output fields: detected (bool), role (cto | vp_engineering | cio | chief_data_officer | head_of_ai | other | none), new_leader_name, started_at, confidence (internal)

detected: false · role: "none" · confidence: 0.2 (insufficient signal from job titles alone)

Classification link: detected == true and role in the named-role enum → Segment 3 (leadership transition, confidence 0.72 for named tech roles, 0.62 for "other"). The LLM is prompted to base judgement on open-role volume as a soft proxy — high senior-engineering role counts are a known correlate of recent leadership changes.

Signal 5 — AI Maturity Scoring (0–3)

Source: agent/enrichment/ai_maturity_scorer.py — Qwen3 via OpenRouter, temperature=0.0.

Scoring rubric (high-weight inputs):

  • [high] ai_adjacent_open_roles — open ML/AI/Data Science/LLM engineer roles
  • [high] named_ai_ml_leadership — VP AI, Head of ML, Chief Data Officer title present
  • [medium] github_org_activity — public GitHub with ML/AI repos
  • [medium] executive_commentary — CEO/CTO public statements on AI strategy
  • [medium] modern_data_ml_stack — PyTorch, LangChain, MLflow evidence
  • [low] strategic_communications — blog posts, talks on AI

Score levels: 0 = no signal · 1 = analytics roles, no AI function · 2 = dedicated AI roles OR named AI leadership · 3 = active AI function + named leadership + open roles + public evidence

score: 0 · confidence: 0.3 · all 6 justifications present with "no evidence found" status

Confidence-to-phrasing mapping (enforced in agent_core.py):

  • confidence < 0.5 OR score == 0 → weak_ai_maturity_signal flag → ask language ("We've seen companies like yours…"), not assert ("You have an AI gap")
  • Segment 4 (capability gap pitch) only unlocks when score ≥ 2 — a hard Python check, not an LLM preference

4. Competitor Gap Brief Status

The competitor gap builder (agent/enrichment/competitor_gap_builder.py) uses Qwen3 via OpenRouter to: (1) identify 5–10 sector competitors from the Crunchbase CSV, (2) score each competitor's AI maturity, (3) compute the prospect's position relative to the sector top quartile. Output is validated against competitor_gap_brief.schema.json before use.

Test prospect: Stripe — competitor_gap_brief.json generated successfully. The pipeline produced a valid brief. Sample output below.

{
  "prospect_domain": "stripe.com",
  "prospect_sector": "Technology",
  "prospect_ai_maturity_score": 0,
  "sector_top_quartile_benchmark": 2.0,
  "competitors_analyzed": [
    { "name": "SG Systems Global", "ai_maturity_score": 1, "top_quartile": true },
    { "name": "Chip Aviation Capital", "ai_maturity_score": 0, "top_quartile": false },
    ...
  ],
  "gap_findings": [
    {
      "finding": "Stripe scores below the sector top-quartile AI maturity benchmark (0 vs 2.0).",
      "signal_type": "capability_gap",
      "confidence": 0.72
    }

}

Limitation observed: The Crunchbase CSV used for competitor identification is a local training dataset, not a live API feed. For Stripe, the competitors returned are small companies in the same CSV region, not actual fintech peers like Adyen or Braintree. This is expected behavior given the data source — the LLM's competitor selection is bounded by what is in the local CSV.

The competitor gap brief populates the gap_findings field in the agent's context, which the agent uses to construct capability-gap arguments in Segment 4 emails (ai_maturity ≥ 2 required to deploy this angle).

5. τ²-Bench Baseline Score and Methodology

5.1 Methodology

The τ²-Bench (tau-squared benchmark) evaluates agent task completion in multi-turn user-simulation environments. The retail domain simulates customer service interactions over a webshop.

  • Harness: eval/tau2_bench_runner.py — wraps tau2-bench's Python API (get_tasks + run_single_task)
  • Model: qwen/qwen3-235b-a22b via OpenRouter, temperature=0.0
  • Slice: dev · 30 tasks · 1 trial per task (facilitator update: 1 trial sufficient for reproduction check)
  • Metric: pass@1 (reward ≥ 0.5 → pass) · Wilson 95% CI · wall-clock p50/p95
  • Trace logging: Each task run emits a Langfuse span and appends to eval/trace_log.jsonl

5.3 Gap Analysis

The reproduction run pass@1 (0.2667) is significantly below the facilitator baseline (0.7267). Two factors account for this:

  1. Single trial vs. five trials: With 1 trial per task, stochastic failures (tool call errors, user simulator mismatches) are not averaged out. 5-trial aggregation substantially improves the mean.
  2. Model difference: The facilitator used qwen3-next-80b-a3b-thinking (a thinking-mode variant with chain-of-thought) against qwen3-235b-a22b which is a larger dense model without native thinking mode. Tool-use accuracy differs meaningfully between these variants on multi-step retail tasks.

6.2 Observations

The competitor gap builder is the dominant latency contributor (30–80s). Two optimization paths for Act III: (1) cache the competitor list per sector (changes slowly), (2) run competitor gap and AI maturity in parallel via asyncio.gather() — currently sequential.

The Playwright job scraper frequently returns no_data (most careers pages block headless browsers). This actually reduces p50 latency (the scraper times out fast for blocked sites) but degrades hiring velocity signal quality.

7.2 What Is Not Working / Limitations

Job Post Scraper — Low Hit Rate. Playwright-based scraping returns no_data for ~75% of tested companies (Stripe, most SaaS companies). Root cause: anti-bot measures on careers pages. Consequence: hiring_velocity.velocity_label is always insufficient_signal, and the Segment 4 AI-role-count gate rarely activates. Mitigation explored but not yet resolved: SerpAPI for public job listings as an alternative source.

τ²-Bench pass@1 Below Facilitator Baseline. Reproduction run scored 0.2667 vs facilitator's 0.7267. Concrete failures: 5 tasks terminated with runner_error (tool call format mismatch with qwen3-235b-a22b); 7 tasks where the agent looped without converging. Not a system fault — the facilitator used a different model variant (thinking mode). The reproduction run is documented as a reference point, not a claimed baseline equivalence.

τ²-Bench Cost Attribution. cost_usd reports $0.0000 for all runs. Root cause: tau2's agent_cost field is populated by tau2's internal cost calculator, which does not have billing data for OpenRouter-routed calls. Actual OpenRouter cost was ~$0.013–0.096 per task based on dashboard, consistent with the facilitator's $0.0199/task baseline. This is a tau2 instrumentation limitation, not a zero-cost run.

SMS Delivery Not Tested End-to-End. Africa's Talking is configured in sandbox mode. No real SMS delivery has been tested because the warm-lead workflow (reply to email → trigger SMS follow-up) has not been activated — it requires a real prospect to reply to an outreach email. The send path and inbound webhook handler are implemented and unit-verifiable but lack an end-to-end delivery trace.

Crunchbase CSV Coverage Gaps. Many well-known companies (Stripe, Notion, Linear) are absent from the training CSV or have partial records. The enricher falls back to no_data, which means the pipeline relies entirely on LLM inference for firmographics in those cases. A live Crunchbase API integration would resolve this but is outside the current budget scope.
